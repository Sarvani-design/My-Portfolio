Design Decisions

The thinking behind the choices

Why start from a reference site instead of a blank prompt?

Vibe coding needs an anchor

A reference restaurant website gave Claude Code a proven layout to adapt, instead of inventing information architecture from a blank page.

Why source data from Google Maps instead of writing copy?

Real data makes it a real pitch

The name, reviews, rating, and address were copied straight from Bun-Wigi's actual Maps listing — grounding the whole site in a real, findable business.

Why not build ordering or booking into the site?

Reuse what's already trusted

The business already delivers through Swiggy/Zomato and takes reservations through Google. Duplicating either would be scope creep with no upside.

Why terracotta + olive instead of a generic food-app palette?

Match the room, not the category

The palette comes from the restaurant's own lantern-lit, wood-and-greenery interior — closer to the actual dining room than a generic red/orange delivery-app look.

Reflection

What this project taught me

  • Real data beats a blank prompt. Starting from an actual Maps listing and a reference template got further, faster, than describing a restaurant from imagination.
  • Getting the aesthetics right took real iteration. The terracotta-and-olive palette and the Anton/Zilla Slab/DM Sans pairing didn't come out of the first prompt — it took several rounds of "warmer," "less rounded," "not that orange" before it felt intentional rather than default vibe-coded output.
  • It hasn't been adopted — and that's real information. I reached out to the actual restaurant; they haven't taken it up yet. Pitching an unsolicited redesign is a different problem than being briefed for one.
  • What I'm proudest of: every CTA works today — a real reservation link, real delivery links, a real map pin.
  • What I'd do differently: get their actual menu file and photo rights before building, instead of working from what's publicly visible on Maps.
